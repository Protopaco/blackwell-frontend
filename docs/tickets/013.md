# [013] Per-Pay-Period Configuration Snapshot (Epic)

**Board status:** Ready (epic — tracked as a group, broken into tickets 024-029)
**Migrated from:** old `docs/TICKETS.md` "Per-pay-period employee roster" note (noted 2026-07-21, raised while building old #11.6). Expanded from employee-only scope to a full epic 2026-07-23.

## Background

**This is `blackwell-backend` work**, tracked here per the decision to keep one ticket board across both repos.

Client-wide config (`Employees`, `Activities`, `FundingSources`, `Holidays`, `Settings` — all in the `PayrollConfig` file) is read *live* every time something touches a pay period: at timesheet generation, at payroll/allocation report generation, and on the status page. There is no point where a pay period's configuration gets fixed in time. Client config is meant to drift as the client's setup evolves; a pay period is a historical record of one slice of time. Today those are entangled — a config change made today silently reaches backward into old, possibly already-closed pay periods.

Concretely confirmed in code: `generateTimesheets.ts`, `getTimesheetStatuses.ts`, and `generatePayrollReport.ts` all independently call `readPayrollConfig`/`getEmployees` live, with no per-period snapshot anywhere. The payroll report file itself (`PayPeriod.payrollReportFileId`) isn't even created at pay-period creation — it's created lazily on first "Generate Payroll Report" click.

**Goal:** at pay period creation, capture ALL configuration values needed to generate and display that period as an explicit stored snapshot — not partial, not read live afterward. Once created, a pay period owns its own copy of its configuration, independent of what client config looks like later. Editing a pay period's data afterward means editing its own snapshot, never the main client config bleeding through.

## Scope

Five categories get captured: **Employees, Activities, FundingSources, Holidays, Settings**. Supervisors are on hold (relic of an earlier iteration). TimesheetFolders excluded (Drive routing, not payroll data).

**Storage**: new tabs in the payroll report workbook (`Employees`, `Activities`, `FundingSources`, `Holidays`, `Settings`), mirroring the same columns as their `PayrollConfig` counterparts — copy-paste of the existing pattern, not a new shape. Report workbook is created at pay-period-creation time (not lazily) specifically so these tabs exist from day one.

Broken into:
- [024] Capture full config snapshot at pay period creation
- [025] Redirect timesheet/report generation reads to the snapshot
- [026] Bulk pay-period-config read endpoint
- [027] Edit Employees within a pay period
- [028] Edit Activities & Funding Sources within a pay period
- [029] Edit Holidays within a pay period

024 and 025 are the actual bug fix and ship together — the snapshot only matters once nothing reads live config anymore. 026-029 build on top.

## Decisions

Resolved 2026-07-23, via requirements discussion:

- **Employees**: captured in full (id, pay rates, `timesheetFileId`, etc.), filtered to `Active` at capture time — but the `Status` column is kept in the snapshot anyway (costs nothing, allows reusing the same DTO). Presence in the roster snapshot *is* roster membership; no separate "included in this period" flag needed — the existing `includeInPayroll` checkbox on the timesheet itself already covers "signed but don't count this."
  - **Add**: allowed anytime (existing mechanic for generating a timesheet for an employee added after others were already generated already covers this).
  - **Remove**: allowed only if that employee doesn't yet have a generated timesheet for the period. Once a timesheet exists, it's never deleted — use `includeInPayroll` instead.
- **Activities & FundingSources**: two-tier lock, because `generateTimesheets.ts` bakes Activity presence into the timesheet's actual rows/columns at generation time (same mechanism as Holidays, see below) while percentages/rates only affect report/allocation math which is recomputed fresh on every regeneration.
  - **Presence** (which Activities/FundingSources exist) locks at **first timesheet generation** for that pay period. FundingSource presence locking at the same point as Activity presence means an Activity can never structurally reference a FundingSource missing from the snapshot.
  - **FundingSource percentages** (and other allocation-only fields) on an Activity stay editable through `Processed`, locking at **allocation report generation** (`Allocated` status) — matching `closePayPeriod.ts`'s existing requirement that an allocation report must exist before closing.
- **Holidays**: captured at pay period creation; timesheet generation reads holiday info from the snapshot, not live config, since holiday designation affects which cells count as holiday hours in the timesheet layout — baked in once, never recomputed. **Locks at first timesheet generation** (there's an edit window between pay-period creation and first timesheet generation). Gets its own tab/edit surface, not just a locked display.
- **Settings** (pay period interval, timesheet template): captured at pay period creation, never editable afterward — it describes how the period itself was constructed, not its contents.
- **Read redirection**: ALL timesheet generation reads (employees, activities, holidays) pull from the pay period's own snapshot, never live client config, once this ships.
- **No existing pay periods to migrate** — pre-launch, so no backfill concern.
- **UI**: one tab per concept (Employees, Activities, FundingSources, Holidays) on the Pay Period page; Settings has no edit tab (locked, informational only if shown at all). Frontend work is a separate epic, not scoped here.
- **Bulk read**: one `PayPeriodConfig`-style DTO/service/route doing a single batched read (mirrors `readPayrollConfig.ts`'s batched `readTabs` call and `getClientSummary.ts`'s aggregation pattern), backing all four UI tabs from one API call instead of four.
