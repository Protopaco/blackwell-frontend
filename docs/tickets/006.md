# [006] Employee email format validation and normalization

**Board status:** Backlog
**Migrated from:** old `docs/TICKETS.md` EM.8

## Background

Noted 2026-07-21, expanded same day. The `email` field on `CreateEmployeeDialog`/`EditEmployeeDialog` currently only checks non-empty (`emailRequired` in `useCreateEmployeeForm`/`useEditEmployeeForm`) — no format check. Confirmed email doesn't appear in any other form in the app.

## Scope

- **Frontend**: standard email-format check (e.g. `something@something.tld`) alongside the existing required check, blocking submit with an inline error the same way `emailRequired` does today.
- **Backend**: two separate utilities (they do different things) — `normalizeEmail.ts` (trim + lowercase) and `validateEmailFormat.ts` (format check) — called in sequence in `createEmployee`/`updateEmployee` before persisting: normalize first, then validate the normalized value, matching the existing `validateActivityFundingSources.ts` pattern of a standalone function shared across create/update. Returns a 422 consistent with other validation failures in this API.
- Normalization is backend-only/persistence-time — the frontend does not transform what the user typed, it just sends it as-is.

## Notes

- Names are deliberately **not** case-normalized anywhere (breaks real names like `McDonald`, `O'Brien`) — this ticket's normalization is email-specific, not a general text-normalization pass.
- A longer-term generic schema-driven validation-layer idea was raised while scoping this ticket but explicitly judged not worth building for this alone (see old docs' "Generic schema-driven backend validation" note) — not on the new board, revisit only if enough similar one-off validations accumulate.
