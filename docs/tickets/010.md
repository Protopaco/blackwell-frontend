# [010] Client Accent Color

**Board status:** Backlog
**Migrated from:** old `docs/TICKETS.md` "Per-client accent color" idea (noted 2026-07-09)

## Background

Give each client a distinct color, applied as the accent/primary color once that client is selected (app bar, possibly card accents) — an instant visual cue for which client you're on, catches "wrong client" mistakes.

## Scope — approach not yet decided

Two options on the table:
1. **Stored field** — `clientColor` added to the `Clients` sheet/schema (backend + frontend change). Curatable, but needs manual data entry per client and someone remembering to set it. User has confirmed the schema change itself is low-effort if this path is chosen.
2. **Derived deterministically from `clientId`** — hash → HSL color, constrained saturation/lightness band for legible contrast. Zero backend change, automatic for every client including existing ones, but not hand-curatable.

Discussion leaned toward option 2, but not settled — user wants to understand the hashing approach better before deciding. No urgency: option 2 has no schema dependency, so there's no cost to deciding late; only option 1 would benefit from earlier commitment.

**Low-cost prep already possible in the meantime**: style new components (e.g. `DashboardCard`) using `theme.palette.primary.main` rather than hardcoded colors, so either approach slots in later with no rework.

## Related idea (noted 2026-07-21, not scoped)

Once a per-client base color exists (either approach), derive `EmployeeTimesheetStatusChip`'s status colors from it instead of the current 5 unrelated hardcoded MUI palette colors (`default`/`info`/`warning`/`primary`/`success`) — convert the client's base color to HSL and step lightness/hue evenly across the fixed 5-status progression (`Not Generated → Generated → Submitted → Approved → Complete`) for an ombre effect tied to the client's identity color.
