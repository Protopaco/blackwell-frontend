# [028] Edit Activities & Funding Sources within a pay period

**Board status:** Backlog
**Part of epic:** [013] Per-Pay-Period Configuration Snapshot
**Depends on:** [024] Capture full config snapshot at pay period creation, [025] Redirect generation reads to the snapshot

## Background

`generateTimesheets.ts` bakes which Activities exist into the timesheet's actual rows/columns at generation time — structurally identical to how Holidays get baked in. FundingSource allocation percentages on an Activity, by contrast, only affect payroll/allocation report math, which gets recomputed fresh every time a report is (re)generated.

## Scope

Two-tier edit rule, enforced server-side:

- **Presence** (adding/removing an Activity or FundingSource from the pay period's snapshot) — allowed only until the first timesheet is generated for that pay period. Locking Activity and FundingSource presence at the same point means an Activity can never end up structurally referencing a FundingSource missing from the snapshot.
- **FundingSource percentages / other allocation-only fields on an Activity** — stay editable through `Processed` status, locking once the allocation report is generated (`Allocated` status) — matching `closePayPeriod.ts`'s existing requirement that an allocation report exist before closing.

New service/route pairs targeting the pay period's `Activities`/`FundingSources` snapshot tabs, following the existing per-domain CRUD pattern (parallel to `services/activity/` and `services/fundingSource/` if those exist client-side).

## Open questions

Exact server-side check for "has the first timesheet been generated yet" — likely the same signal [027] uses for its removal guard, should be shared rather than reimplemented.
