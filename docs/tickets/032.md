# [032] Activities tab

**Board status:** Backlog
**Part of epic:** [030] Per-Pay-Period Configuration UI
**Depends on:** [028] Edit Activities & Funding Sources within a pay period (backend)

## Background

New tab on the Pay Period page for the period's Activities snapshot. Per [030]'s decision, this reuses the existing client-wide Activities table and Create/Edit modal components (from `ActivitiesManagement`) rather than building new ones — same UI, pointed at the pay period's endpoints instead of `PayrollConfig`'s.

## Scope

- New tab, new page under `client/:clientId/payPeriod/:payPeriodId`, following the existing tab/route pattern in `PayPeriodLayout.tsx`/`router.tsx`.
- Reuse the existing Activities table + Create/Edit dialog components, wired to the pay period's Activity endpoints ([028]) instead of the client-wide ones.
- Two lock states from [028], both need to surface in this reused UI:
  - **Presence** (add/remove) locks at first timesheet generation for the period.
  - **FundingSource percentages on an Activity** stay editable through `Processed`, locking only at `Allocated`.
- Add flow: since reusing the client-wide "Create" modal creates a brand-new Activity record, confirm whether adding to a pay period should instead be a picker over existing client Activities not yet on this period, or whether reusing "Create" as-is is fine here.
