# [016] Generic Schema Driven Backend Validation

**Board status:** Backlog
**Migrated from:** old `docs/TICKETS.md` "Generic schema-driven backend validation" idea (noted 2026-07-21, raised while scoping EM.8 / [006])

## Background

**This is `blackwell-backend` work**, tracked here per the decision to keep one ticket board across both repos.

Backend validation today is hand-rolled per service (e.g. `validateActivityFundingSources.ts`, one-off checks in `createEmployee`/`updateEmployee`, and the new email format/normalization utilities from [006]). A longer-term alternative is a shared validation layer — define a schema/interface per object (employee, activity, funding source, etc.), run any create/update payload through it, and get consistent field-shape checks (email format, URL format, enum membership, etc.) across every endpoint instead of re-deriving the same kind of check per service. Zod is the natural fit in this Node/TypeScript stack.

Explicitly judged **not worth building for [006] alone** — that ticket's scope (one field, two services) is small enough for a single-purpose utility function, following the existing `validateActivityFundingSources.ts` pattern.

## Scope

Not scoped. Revisit if/when enough services accumulate similar one-off validation that the duplication becomes the actual pain point — this ticket exists to track the idea, not to schedule the work.
