# [013] Per pay period employee roster

**Board status:** Backlog
**Migrated from:** old `docs/TICKETS.md` "Per-pay-period employee roster" note (noted 2026-07-21, raised while building old #11.6)

## Background

**This is `blackwell-backend` work**, tracked here per the decision to keep one ticket board across both repos.

The Timesheet Status page currently lists **every currently-active employee**, not just employees who existed at the time a given pay period ran. `getTimesheetStatuses.ts` (`src/services/timesheet/getTimesheetStatuses.ts:20-21`) joins all `Active` employees against the pay period unconditionally — so an employee added *after* a pay period was created/closed still shows up on it as `NotGenerated`. `Employee` (`src/models/Employee.ts`) also has no `hireDate`/`createdDate` field to derive membership from automatically.

**Confirmed current architecture** (investigated 2026-07-21): there is no per-pay-period roster concept anywhere today — `generateTimesheets.ts` (timesheet generation) and `getTimesheetStatuses.ts` (status page) both independently read the same source, `payrollConfig.employees` filtered to `Active`, with no per-period distinction. The payroll report Sheets file itself (`PayPeriod.payrollReportFileId`) isn't even created at pay-period creation — it's created lazily on first "Generate Payroll Report" click (`generatePayrollReport.ts:80-85`, via `createOAuthWorkbook`), as a bare file with no pre-built tabs.

## Scope — direction discussed, not yet scoped into tasks

Treat pay-period employee membership as an explicit stored roster, the same way Activities/Funding Sources are explicit client-wide config rather than derived — but scoped *per pay period*, not client-wide. Requires:
1. Creating the payroll report file at pay-period creation time (not lazily on first report generation) so a roster tab can exist from day one and gate generation itself, seeded with then-active employees.
2. Redirecting `generateTimesheets.ts`, `getTimesheetStatuses.ts`, and `generatePayrollReport.ts` to read that roster instead of live `Active`-employee queries.
3. A new "add employee to pay period" action (likely from Employee Management) that appends to the *current* pay period's roster after the fact, without altering past/closed pay periods' rosters.

## Open questions

Scope touches pay-period creation timing, three existing read paths, and a new write path — deliberately deferred, not tackled without a dedicated design pass first.
