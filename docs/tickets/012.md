# [012] Caching System for timesheets (Sheets API Quota)

**Board status:** Backlog
**Migrated from:** old `docs/TICKETS.md` "Consider a caching system for timesheets" note (noted 2026-07-21)

## Background

**This is `blackwell-backend` work**, tracked here per the decision to keep one ticket board across both repos.

Raised while discussing recurring Google Sheets API 429/quota errors: opening the Timesheet Status page fires a full read per active employee (`getTimesheetStatuses.ts` calls `readTimesheetDetail` per employee ŌĆö see `generatePayrollReport.ts`'s similar per-employee loop for the Payroll Report page), which is both slow to load and a meaningful chunk of the 60-req/min quota, especially as employee count grows.

## Scope

Idea: some way to answer "do I need to re-read this sheet, or is my cached copy still current?" instead of always re-reading.

**Promising direction not yet explored in code**: the Google **Drive** API (distinct from the Sheets API) exposes a file's `modifiedTime` via a lightweight `files.get(fileId, fields: 'modifiedTime')` call ŌĆö cheap enough to poll before deciding whether a full Sheets read is needed, since a timesheet not edited since the last read has nothing new to fetch.

## Open questions

- Where would the cache live ŌĆö in-memory per backend process, or something more durable?
- What does "current" mean across concurrent requests?
- Does the `modifiedTime` check itself burn enough quota to still help at scale?

## Decisions

Resolved 2026-07-22, based on investigation of the current `blackwell-backend` codebase.

- **Direction confirmed**: use Drive API `files.get(fileId, fields: 'modifiedTime')` as a cheap pre-check before a full Sheets read. No new client needed ŌĆö a service-account Drive client and a separate `driveLimiter` (Bottleneck, 60/min) already exist and are unused for this purpose today.
- **Cache location**: in-memory per backend process, following the existing `createCache<T>` TTL-cache pattern (`src/utils/cache.ts`) already used for `clientsCache`, `payrollConfigCache`, `currentHoursCache`, etc. Deployment is confirmed single-instance (Railway, per `docs/ARCHITECTURE.md`), so a shared/durable store would be solving a problem that doesn't exist yet. Add a "known limitation: per-process, not shared across instances" note, matching the existing caches' documented limitation, in case of future horizontal scaling.
- **Concurrency**: each employee has their own `timesheetFileId`, so there's no cross-employee race. The real race is two overlapping requests for the *same* employee's timesheet (e.g. status page load overlapping report generation) both missing the cache and firing duplicate reads. Mitigate with request coalescing ŌĆö an in-flight-promise map keyed by `timesheetFileId`, so a second caller awaits the first's in-progress fetch instead of re-reading.
- **Quota math nets out ahead**: Drive quota is a separate bucket from the 60/min Sheets quota (confirmed ŌĆö distinct rate limiters), so every cache hit converts up to 2 Sheets calls (`readManifest` + `readTabValues` inside `readTimesheetDetail`) into a Drive call that doesn't compete for the scarce Sheets budget at all. This is a net win for Sheets quota regardless of hit rate; the only downside of a miss is one extra network round trip, not quota.
- **Invalidation**: `modifiedTime` alone is sufficient ŌĆö no extra invalidation hook needed. This backend never writes signature/hours/`includeInPayroll` data to timesheet workbooks after initial generation (`generateTimesheets.ts` is the only writer, and that's a one-time initial population); all signing/approval happens directly in the Sheets UI, outside this API. Any edit ŌĆö whether from this app or a human in the sheet ŌĆö bumps Drive's `modifiedTime` regardless of who made it, so it's a reliable staleness signal on its own. (Unlike `currentHoursCache`, which needs a manual invalidation call because its write path doesn't naturally correlate with what it checks.)

Ready to move to implementation once this ticket is picked up off the board.
