# [005] Create Client Management Workflow

**Board status:** Backlog
**Migrated from:** old `docs/TICKETS.md` "Client Administration" epic

## Background

No frontend surface exists yet for creating a client or editing client identity/status. Backend already supports `GET`/`POST /api/v1/client` and `PUT /api/v1/client/{clientId}`.

Backend client creation is heavier than other creates — it provisions real Drive/Sheets infrastructure: creates the Payroll Config and Pay Period Registry files, creates required config tabs, and seeds Settings. There is deliberately no delete flow; deactivation via `status` is the only supported removal path (matches the pattern used elsewhere — e.g. Timesheet Folder Management).

## Scope

Build the frontend surface for:
- Creating a client (identity fields + whatever Drive/Sheets provisioning inputs the backend create endpoint needs)
- Editing client identity/status (activate/deactivate, no hard delete)

## Open questions

- UX for existing-folder-link vs. create-new-folder choices during client creation — the backend deliberately does **not** silently create or roll back partial Drive setup on failure, so the frontend needs to communicate partial-failure states clearly rather than assuming atomicity.
- Where does "Create Client" get triggered from, given there's no client-selected yet at that point (no `ClientSummary` to put a button on)?
