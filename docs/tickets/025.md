# [025] Redirect timesheet/report generation reads to the snapshot

**Board status:** Backlog
**Part of epic:** [013] Per-Pay-Period Configuration Snapshot
**Depends on:** [024] Capture full config snapshot at pay period creation

## Background

Three services currently read client-wide `PayrollConfig` live instead of a per-period snapshot:

- `generateTimesheets.ts` (`src/services/timesheet/generateTimesheets.ts:65-69`) — reads `payrollConfig.employees` filtered to `Active`, plus `payrollConfig.activities` and `payrollConfig.holidays`, to build timesheet layout.
- `getTimesheetStatuses.ts` (`src/services/timesheet/getTimesheetStatuses.ts:20-21`) — reads all active employees via `getEmployees(clientId)` unconditionally, so an employee added after a pay period was created still shows up on it.
- `generatePayrollReport.ts` (`src/services/payrollReport/generatePayrollReport.ts:42-50`) — reads `payrollConfig.employees` and `payrollConfig.activities` live.

## Scope

Redirect all three to read from the pay period's own snapshot tabs (created in [024]) instead of live `PayrollConfig`. This is the actual fix for the drift bug — an employee/activity/holiday added to client config after a pay period was created no longer affects that pay period.

Once this ships, together with [024], a pay period's generated timesheets and reports are fully independent of subsequent changes to client-wide config.

## Out of scope

Editing the snapshot after creation — that's [027]/[028]/[029]. This ticket only redirects existing reads to the already-captured data.
